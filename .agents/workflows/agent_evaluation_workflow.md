# LLM Agent 测试-评估-优化 工作流指示

由于系统的核心控制权和计算节点由大模型（LLM）构成，传统的单元测试和自动化脚本只能捕获崩溃和类型错误，无法有效衡量系统在“常识推理”、“工具选择”、“计划分解层级”以及“最后一公里触达”等维度的软性智能表现。

因此，在每次架构、Prompt、工具集或核心组件发生重大更新后，必须严格遵守以下人工（或 AI Agent亲自）驱动的主观评估与优化工作流。

## 阶段一：准备与全量回归测试 (Test)

1. **构建与环境对齐**: 确保所有的代码修改已被编译 (`cargo build`)，并且 Daemon 已基于最新二进制重启。
2. **执行基准用例集**: 运行标准的测试脚本（例如 `./run_tests.sh`），该脚本应包含能够覆盖所有 Agent 类别（Research, Coder, Architect, General）和所有工具集的多个用例。包含但不限于：
   - 信息检索与事实核查（依赖联网和地理时间信息）
   - 代码编写与文件系统交互
   - 跨阶段长流转任务（如撰写复杂的旅行计划）
   - 极端或故意模糊的用户指令
3. **数据沉淀**: 测试结束后，收集所有生成的日志与 Traces 数据源（例如 `test_traces/` 目录下的 CLI 日志和 JSON Traces）。

## 阶段二：深度专家主观评估 (Evaluate)

此阶段**必须由 AI 亲自阅读原始日志流**（而不仅是依赖 Python 脚本出具的统计摘要），进行深度的专家诊断。请重点关注并记录：

1. **核心衡量维度**:
   - **完成度与准确率**: 用户的核心目的有没有被“精准满足”（不缺漏、不冗余）？
   - **执行路径 (Trace Path)**:
     - 工具调用是否高效？是否出现了死循环或疯狂重试？
     - “最后一公里”：Agent 是否将最终结果自然、准确地传达给了用户，而不是只发了一句“执行成功”或代码状态码？
     - 基础常识：是否对客观物理世界（时间、位置、语言）产生幻觉？
   - **路由与评估 (Router QA)**: Router 的 QA 机制是否过严导致无限打回？或过松导致残缺结果流出？

2. **对比历史（前后回归分析）**:
   - **老问题是否改善**：明确指出上一个优化周期（Iteration）中尝试修复的问题（例如：缺少位置上下文导致的反复询问）是否已经消失。
   - **是否引入新问题**：检查本次修改是否造成了预期外的负面副作用（例如：注入了上下文，但导致 Token 长度过长产生截断报错）。

3. **撰写详细评估报告**: 
   将发现归档为标准的评估报告结构，保存到项目根目录下的 `reports/` 文件夹中（如 `reports/agent_evaluation_iteration_X.md`），必须包含：**整体执行成功率变化**、**对比上次的显著改善**、**残留或新暴露的缺陷**。
   另外，**对于评估的典型用例，必须逐一输出以下明细框架，以便用户共同评审**：
   - **用户输入 (User Input)**: 原始的请求内容
   - **给用户的返回 (Return to User)**: Agent 最终输出给用户的回答内容
   - **结果评分 (Result Score)**: 对输出是否真正解决了输入问题的客观评分（例如 1-10 分或 Pass/Fail）及简要理由
   - **过程评分 (Process Score)**: 评价维度包括思路是否清晰、动作是否简洁有效（没有疯狂重试或多余工具调用）

## 阶段三：制定优化方案 (Optimize)

基于上述产出的评估报告，制定具有明确行动点（Action Items）的优化计划：

1. 对于 Agent，这涉及决定修改哪个具体文件（如 Prompt 模板、DAG 引擎的数据流、路由权重、工具库解析器）。
2. 在正式改动代码前，产出类似 `implementation_plan.md` 的任务列表，以防止修改偏离主线。

## 阶段四：持续迭代 (Re-Test & Re-Evaluate)

完成代码调整后，将当前状态作为新的基准（Baseline），自动无缝进入下一轮的“阶段一”。形成飞轮闭环。

---

**给 AI Agent 的执行建议**：
当被要求“测试并评估”时，请使用 `run_command` 执行脚本，等待完成。然后使用 `grep_search` 或 `view_file` 深入探查具体的日志文件，绝对不要仅满足于 `analyze_traces.py` 这种弱脚本的结论！必须展示你的专家级系统分析能力。
