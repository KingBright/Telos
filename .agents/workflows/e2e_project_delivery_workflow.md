# 端到端项目交付评估与优化工作流 (E2E Project Delivery & Optimization Workflow)

**“以终为始，以项目成功为最高目标。系统绝不能仅仅‘看似很强’，无法交付最终可用结果的项目就是最大的失败。”**

本工作流旨在通过执行不同梯度的真实项目测试用例，强制系统在没有人工干预的情况下独立完成所有工作。核心目标是暴露系统在面对异常时的应对能力，并借此驱动架构的持续进化与漏洞修复。

## 阶段 一：动态构建全新场景端到端自动化压测

为了避免低水平的重复执行，**严禁**反复使用之前硬编码的测试脚本文案。每次执行此工作流时，负责操作的 AI 必须先依靠推理**动态生成 3 个全新的、涵盖未测试领域**的项目创意，并将它们作为 payload 写入临时 Python 执行脚本。确保所有测试都是在禁用 `approval`（免打扰）的模式下进行。

1. **重建并重启最新的 Telos 守护进程**
   执行基础安装脚本，确保我们测试的是包含最新补丁（例如：泛化 IntegrationTester 架构等）的系统版本。
// turbo
2. bash install.sh
   
3. **动态生成并执行用例：基础结构与非代码能力测试 (Case 1)**
   - **AI 动作**：思考一个全新的纯文档/配置文件项目创意（例如：AWS Terraform 配置说明模板、开源协议律师免责声明库等）。
   - **执行**：将其写入 `tmp_case_1_static.py` 的 payload 中并运行。验证系统能否摆脱编译依赖，不触发无意义的报错。
// turbo
4. python3 tmp_case_1_static.py

5. **动态生成并执行用例：中等复杂度带依赖的完整项目投产 (Case 2)**
   - **AI 动作**：构思一个冷门的底层 Rust 命令行程序，必须依赖不同类型的第三方库（例如：系统监控 `sysinfo` 读取 CPU 频率、`zip` 解压缩归档分析器、`imagesize` 批量检测脚手架等）。
   - **执行**：写入 `tmp_case_2_cli.py` 的 payload 并运行。验证从架构师分解清单到底层 Worker 编写正确代码的顺畅度。
// turbo
6. python3 tmp_case_2_cli.py

7. **动态生成并执行用例：高复杂度异步或跨线程通信异常注入 (Case 3)**
   - **AI 动作**：设计一个极易引发架构师规划遗漏或 Worker 生命周期借用报错的高复杂度应用（例如：基于 `tokio-tungstenite` 的 WebSocket 中继服务器、包含 `mpsc::channel` 和多线程共享 `Arc<Mutex>` 状态的数据聚合器）。
   - **执行**：写入 `tmp_case_3_async.py` 的 payload 并运行。故意探测复杂代码报错下，第一道防线与第二道防线（`IntegrationTester` 与 `TechLead`）的真实异常捕捉和闭环表现。
// turbo
8. python3 tmp_case_3_async.py

## 阶段 二：系统应对能力深度评估 (System Capability Assessment)

不要停留在“是否抛错”的表面现象，对于每个测试用例，请按以下维度进行深度诊断：

1. **执行流畅度评估：** 观察日志，检查节点流转中是否有不必要的挂起或理解误差。架构师分配的具体领域指令（Language/Framework）是否被下游 Worker 准确遵守？
2. **异常捕获与反弹评估：** 
   - 当 Worker 给出无效代码或语法错误时，第一道防线（`HarnessValidatorNode`）是否能正确挂起并打回重作？
   - 当面临更为隐蔽的逻辑借用冲突时，第二道防线（`IntegrationTesterNode`）是否成功抛出？
   - `TechLead`（技术负责人代理）被召唤时，是否能真正根据 `cargo check` 返回的具体错误，精准下发文件级的闭环 Patch 修复，而不是陷入废话或者无关修改？
3. **架构健壮性判断：** 出现预料外情况时（例如依赖网络下载断开、大模型返回损坏的 JSON），DAG 引发的是优雅终止保护（Graceful Abort）还是灾难性雪崩？

## 阶段 三：“以终为始”的交付结果核验 (Outcome-Driven Verification)

项目的唯一成败取决于交付的终态是否可用。脱离结果谈 AI 进展是没意义的。

1. **检查最终产品库**：前往 `~/.telos/projects/<project_name>` 目录下。
2. **核对制品要求**：
   - 对于非代码项目（Case 1）：验证是否的确只有目标 MD 文件，内容是否准确涵盖设计文档要素？
   - 对于代码项目（Case 2 & 3）：进入目录，直接执行对应的原生命令（如 `cargo run` 或 `npm start`）。
3. **验收终态价值**：程序是否能按预期运行？是否拥有完整的 `Cargo.toml` 结构？业务逻辑有无缺失？

## 阶段 四：闭环迭代优化 (Iterative Optimization)

如果在第二阶段（异常应对弱）或第三阶段（交付不可用）中发现任何短板，必须制定并立刻执行纠错优化方案：

1. **识别弱点层次**：
   - 是系统级短板？（例如框架里存在特定的硬编码影响泛用性）
   - 是代理级短板？（例如 `TechLead` 或 `Worker` Prompt 限定太弱，导致经常出现未闭合的 block）
   - 是引擎级短板？（例如记忆库没有正确提供相关的 Context 导致库版本混乱）
2. **微调与架构翻新**：根据识别的弱点提出架构重构计划或系统 Prompt 升级。
3. **重复执行本工作流**，直到所有的系统更新能够让三个测试用例实现 100% 的无人干预成功交付。
